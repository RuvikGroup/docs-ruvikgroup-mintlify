# Issue Template

Use this template when creating issues in GitHub. Copy the markdown below into your issue description.

<Note>
  GitHub will automatically use this template if you save it as `.github/ISSUE_TEMPLATE/bug_report.md` in your repository.
</Note>

```markdown
## Issue Type
- [ ] Bug
- [ ] Feature Request
- [ ] Question
- [ ] Other (please specify)

## Priority
- [ ] Low
- [ ] Medium
- [ ] High
- [ ] Critical

## Subject of the issue
[Provide a clear and concise description of the issue]

## Your environment
* Production or Staging: 
* Version of the app: 
* Device (if applicable): 
* If web, browser and version: 

## Account Information
* Username: 
* Password: 
[Note: If sensitive, please provide a way for the team to securely access this information]

## Steps to reproduce
[Describe how to reproduce the issue. A video is highly recommended. You can also use bullet points:]
* Step 1
* Step 2
* Step 3

## Expected behaviour
[Describe what should happen]

## Actual behaviour
[Describe what actually happens]

## Attachments (Screenshots, Videos)
[Attach any relevant screenshots or videos that help illustrate the issue]

## Additional context
[Add any other context about the problem here]

## Possible Solution
[If you have suggestions on how to fix the issue, please describe them here]
```

## Issue Types Explained

### Bug Report
Use for reporting errors, crashes, or unexpected behavior:
- Application crashes
- Features not working as documented
- Performance issues
- Security vulnerabilities

### Feature Request
Use for suggesting new functionality:
- New features or capabilities
- Improvements to existing features
- Integration requests
- UI/UX enhancements

### Question
Use for:
- Clarification on functionality
- Documentation questions
- Implementation guidance
- Best practices

## Priority Guidelines

<Warning>
  Only mark issues as **Critical** if they affect production systems and require immediate attention.
</Warning>

### Priority Levels

1. **Low**: Minor issues with workarounds available
   - Cosmetic issues
   - Minor documentation updates
   - Nice-to-have features

2. **Medium**: Important but not urgent
   - Bugs affecting some users
   - Performance improvements
   - Feature requests with clear value

3. **High**: Significant impact on users
   - Bugs affecting many users
   - Security issues (non-critical)
   - Blocking features

4. **Critical**: Requires immediate attention
   - Production outages
   - Data loss risks
   - Critical security vulnerabilities
   - Complete feature failures

## Best Practices

1. **Search First**: Check if the issue already exists
2. **One Issue Per Report**: Don't combine multiple issues
3. **Be Specific**: Include exact error messages
4. **Provide Context**: Include relevant logs or console output
5. **Follow Up**: Respond to questions from maintainers

## Security Issues

<Warning>
  For security vulnerabilities, do NOT create a public issue. Instead, email security@ruvikgroup.com directly.
</Warning>
